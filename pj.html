알고리즘 전략의 적절성

1. 선택한 알고리즘 전략

완전 탐색 (Exhaustive Search / Brute Force Algorithm)

모든 가능한 경로를 빠짐없이 탐색하여 최적해를 찾는 방법

가능한 모든 순열을 생성하고, 각 순열에 대해 총 이동거리를 계산하여 최소값을 선택

2. 왜 TSP 문제에서 Brute Force Algorithm이 사용되어야 하는가?

2.1 정답의 무결성 (Mathematical Integrity)

대부분의 TSP 해결 알고리즘은 연산 속도를 위해 정확도를 희생하는 근사해를 찾습니다. 이는 "적당히 괜찮은 경로"라고 타협하는 방식입니다.

반면, Brute Force Algorithm은 타협하지 않습니다.

가능한 모든 경우의 수를 계산하므로 수학적으로 100% 증명된 해를 보장합니다.

실제 적용: 응급 의료 이송, 정밀 제어와 같이 오차가 허용되지 않는 무결점 시스템 구축에 필수적입니다.

2.2 소규모 데이터에서의 압도적인 가성비

방문 지점(Node)의 개수(n)가 적을 때 컴퓨터의 연산 속도가 워낙 빨라, 복잡한 휴리스틱 알고리즘을 설계하는 시간보다 BFA로 돌리는 게 개발 비용 대비 효율이 훨씬 좋습니다.

노드 수 탐색 경로 수 실행 시간 평가
5개 24 0.001초 ✓ 즉시
8개 5,040 0.1초 ✓ 실용적
10개 362,880 1-2초 △ 허용

2.3 다른 알고리즘의 '정답지' 역할 (Benchmark Standard)

누군가 "획기적인 알고리즘을 개발했다"고 주장하려면, 그 알고리즘이 찾은 경로가 진짜 최단 경로인지 확인할 방법이 있어야 합니다.

이때 BFA로 구한 '진짜 정답'이 있어야 검증이 가능합니다.

다른 알고리즘의 성능을 평가하는 기준점(Baseline) 역할을 합니다.

2.4 구현의 단순함과 유지보수성

복잡한 알고리즘은 예외 처리가 까다롭고 버그가 발생할 확률이 높습니다.

반면 BFA는 로직이 매우 단순합니다:

  1. 모든 순열 생성
  2. 각 순열의 거리 계산
  3. 최소값 선택

이러한 단순함은 코드 검증, 디버깅, 유지보수를 훨씬 쉽게 만듭니다.

3. 실생활에서 쓰이는 Brute Force Algorithm

Amazon의 라스트 마일 전략 (Last Mile Routing)

세계 최대 물류 기업인 Amazon 역시 거대 규모의 배송 경로 문제를 해결하기 위해, 전체를 한 번에 계산하는 대신 문제를 작게 나누는 전략을 사용합니다.

📦 구역화 (Clustering)

Amazon의 라스트 마일 연구 데이터셋에 따르면, 각 배송지에는 Zone ID(지리적 계획 구역)가 부여되어 있습니다.

이는 도시 전체를 하나의 문제로 보지 않고, 배송 구역을 동네나 단지 단위의 작은 클러스터로 나누어 관리함을 시사합니다.

🎯 소규모 최적화

이렇게 쪼개진 작은 구역(Cluster) 내부에서는 방문해야 할 노드의 수가 현저히 줄어듭니다.

노드 수가 제한된 환경(Small Scale Nodes)에서는 복잡한 근사 알고리즘보다 정확한 연산(Exact Algorithm)이 훨씬 효과적이며 신뢰성이 높습니다.

MIT 운송 물류 센터(MIT CTL)의 접근

물류 최적화의 정석으로 불리는 Cluster-First, Route-Second 전략 역시 이와 맥락을 같이합니다.

  1. Cluster-First: 대규모 문제를 클러스터링을 통해 소규모 문제로 변환
  2. Route-Second: 각 그룹 내에서 최적의 경로를 도출

우리 프로젝트는 이 중 Route-Second 단계, 즉 확정된 소규모 구역 내에서의 완벽한 경로 탐색에 집중하여 BFA의 효용성을 극대화했습니다.

참고: MIT CTL Research Paper

알고리즘 로직이 직관적이고 이해하기 쉽습니다

  • 순열 생성 → 거리 계산 → 최소값 선택의 명확한 흐름
  • 디버깅과 검증이 용이

✅ 장점 3: 소규모 문제에 최적

알고리즘 설계 및 구조

1. 해결 과정 개요

2. 알고리즘 구조 설명

핵심 함수 구조

// 1. 거리 계산 함수 function calculateDistance(point1, point2) { dx = point2.x - point1.x dy = point2.y - point1.y return sqrt(dx² + dy²) // 유클리드 거리 } // 2. 경로 총 거리 계산 function calculatePathLength(path, points) { totalDistance = 0 for i = 0 to path.length: start = points[path[i]] end = points[path[(i+1) % path.length]] totalDistance += calculateDistance(start, end) return totalDistance } // 3. TSP 해결 (핵심 로직 - 완전 탐색) function solveTSP(points) { startPoint = points[0] // 물류센터 고정 remainingPoints = points[1..n] minDistance = INFINITY bestPath = null pathsExplored = 0 // 모든 순열 탐색 (Brute Force) for each permutation in permutations(remainingPoints): currentPath = [startPoint] + permutation currentLength = calculatePathLength(currentPath, points) pathsExplored++ if currentLength < minDistance: minDistance = currentLength bestPath = currentPath return {bestPath, minDistance, pathsExplored} }

순열 생성 방법

// Generator 함수를 사용한 효율적 순열 생성 function* permutations(array) { if array.length <= 1: yield array else: for each element in array: remaining = array without element for each perm in permutations(remaining): yield [element] + perm }

5. 핵심 로직 상세 설명

🎯 완전 탐색 전략 적용 부분

Step 1: 탐색 공간 축소

시작점을 고정함으로써 N! → (N-1)! 로 탐색 공간 감소

const startPoint = allIndices[0]; // P0 고정 const remainingPoints = allIndices.slice(1); // P1, P2, ..., Pn

Step 2: 순열 생성 및 평가

Generator를 사용하여 메모리 효율적으로 모든 순열 생성

for (let perm of permutations(remainingPoints)) { const currentPath = [startPoint].concat(perm); const currentLength = calculatePathLength(currentPath); pathsExplored++; // 더 짧은 경로 발견 시 업데이트 if (currentLength < minDistance) { minDistance = currentLength; bestPath = currentPath; } }

Step 3: 폐경로 거리 계산

마지막 노드에서 시작점으로 돌아오는 거리까지 포함

// 경로의 각 구간 거리 합산 for (let i = 0; i < path.length; i++) { const start = points[path[i]]; const end = points[path[(i + 1) % path.length]]; // % 연산으로 순환 total += calculateDistance(start, end); }

4. 개선 방안 및 대안

단기 개선 방안

1. 가지치기 (Branch and Bound)

현재 경로가 이미 최소 거리를 초과하면 더 이상 탐색하지 않음

if (currentPartialDistance >= minDistance): continue // 이 경로는 더 탐색하지 않음

예상 효과: 평균 50-70% 탐색 공간 감소

2. 거리 행렬 사전 계산

모든 노드 쌍 간 거리를 미리 계산하여 저장

distanceMatrix[i][j] = calculateDistance(points[i], points[j]) // 이후 O(1)에 거리 조회 가능

예상 효과: 거리 계산 시간 50% 감소

3. Generator 사용 (✅ 이미 적용)

순열을 한 번에 생성하지 않고 필요할 때마다 생성

효과: 메모리 사용량 O(n!) → O(n)로 감소

장기 대안 알고리즘

알고리즘 시간 복잡도 적용 범위 정확도
동적 프로그래밍 O(n²·2ⁿ) N ≤ 20 100% 최적해
분기 한정법 O(n!)~O(n²·2ⁿ) N ≤ 30 100% 최적해
Christofides O(n³) N ≤ 1000 1.5배 이내 보장
유전 알고리즘 O(g·p·n²) N > 100 근사해

  • 갈등 해결: 의견 차이를 건설적으로 해결하는 경험

2. 아쉬운 점

개선이 필요한 부분

  • 확장성 제한: 노드 수가 10개를 초과하면 실용성이 급격히 떨어짐
  • 고급 알고리즘 미적용: 동적 프로그래밍이나 분기 한정법 등 더 효율적인 방법 미구현
  • 실제 지도 연동 부재: 실제 도로 네트워크를 고려하지 못함
  • 다중 출발지 미지원: 여러 물류센터가 있는 경우를 다루지 못함
  • 모바일 최적화: 스마트폰에서의 사용성이 다소 부족

3. 알고리즘 전략의 실제 의미

🔍 완전 탐색이 가르쳐준 것

완전 탐색은 "모든 가능성을 확인하는 것"의 가치를 보여줍니다. 비효율적으로 보일 수 있지만, 작은 문제에서는 가장 확실한 방법이며, 다른 알고리즘의 정확성을 검증하는 기준이 됩니다.

이는 실생활에서도 중요한 교훈입니다. 때로는 빠른 답보다 확실한 답이 필요하며, 모든 선택지를 신중히 검토하는 것이 최선의 결과를 가져올 수 있습니다.
